2) payload 的思路：破坏原来的条件并插入“总是为真的条件”

我们想让 SQL 的 WHERE 子句变成总是为真，从而绕过密码检查。步骤如下：

  • admin' —— 这个单引号用于结束原来 username 的开引号。原来是 username = '...，我们用一个单引号把字符串闭合，脱出字符串上下文进入 SQL 语句层面。
  • and 1=1 —— 在脱出的地方插入一个条件 AND 1=1。1=1 恒为真，不会影响查询的“真/假”判断。
  • # —— 在 MySQL 中 # 是注释符，后面的内容会被忽略（在其他数据库中常用 -- 或 /* ... */）。这个注释把后面原本拼接的 AND password = '...' 或其他剩余片段注释掉，防止语法出错或再次加约束。

把它拼回去，想象最终 SQL 变为：

# 后面被忽略，所以实际上条件就是 username = 'admin' and 1=1 → 等同于 username = 'admin'。如果数据库里有 admin 这条记录，查询会返回行，登录被绕过。

3) 为什么用 1=1？为什么 1=2 会失败？

  • 1=1 恒为真，所以加上去不会让 WHERE 变成假；如果用户名匹配到行，就返回该行 → 绕过密码检查。
  • 1=2 恒为假，那么 username = 'admin' AND 1=2 总是假（无论 username 是否为 admin），查询不会返回任何行，登录失败。
    用 1=1/1=2 是布尔盲注里常用的简单试验：把条件换成真/假的子句，观察响应是否不同，从而判断是否存在注入点。

4) 注释符为什么必要？

因为原查询后面通常还有 AND password = '...'。如果你仅写 admin' and 1=1 而不注释掉后面的内容，最终 SQL 可能变成：

这会造成语法错误（多了一个单引号），所以用注释把后面剩余的 SQL 去掉。不同 DB 的注释符：

  • MySQL: # 或 -- （-- 后通常需要一个空格）或 /* ... */
  • PostgreSQL/Oracle: --、/* ... */
    注意 -- 必须接空格或换行；有时用 /*...*/ 更稳妥。

5) 常见变体（都是同一思路，只是写法不同）

  • admin' OR '1'='1' -- （把条件写成 OR）
  • admin'/**/OR/**/'1'='1'/* （用注释块/空白绕过简单过滤）
  • admin'%0aOR%0a'1'='1'-- （用换行/URL 编码绕过词过滤）
  • admin' and (select 1 from dual where 1=1)=1 # （更复杂的布尔表达式）

AND 与 OR 的选择取决于你想改变原查询逻辑的方式（插入 OR '1'='1' 会使整个 WHERE 条件总为真；插入 AND 1=1 是另一种等价手段）。

6) 布尔盲注检测方法（为什么用 1=1 / 1=2）

当页面对返回结果没有直接展示数据库数据（没有错误或明显内容差异），但对“是否成功”或页面长度/响应时间有差异时，可以用布尔盲注：

  • 发送 username = admin' AND 1=1 #，记录响应（比如页面长度、状态、是否跳转）。
  • 发送 username = admin' AND 1=2 #，对比响应。
  • 如果两次响应不一样，说明注入成立（后端确实把你的条件纳入 SQL）。这就是“布尔盲注”的核心：通过真假条件的对照来获知数据库行的存在/字段值。
